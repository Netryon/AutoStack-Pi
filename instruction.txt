You are building a full interactive installer for a Raspberry Pi 3B+ self-hosted stack.

Your job is to create an installer that prepares and deploys the system in a controlled sequence.

The installer must:
1. Run step by step.
2. Prompt the user for all variable values as needed while progressing.
3. Create all directories, compose files, env files, scripts, and config files.
4. Avoid hardcoding any value that may change in the future.
5. Deploy services only after preparation is complete.
6. Start containers one by one at the end.
7. Verify each container is running successfully.
8. Log important installer actions.
9. Clean up old Docker images/logs over time to protect the 32 GB SD card.
10. Harden the firewall at the end so that only SSH and intended service ports remain open.

Important design rule:
Do not just output static YAML and env files.
Instead, create an INSTALLER that generates and fills those files interactively.

The installer must ask me for values during execution, such as:
- timezone
- DuckDNS token
- DuckDNS subdomains
- Pi-hole admin password
- Pi-hole upstream DNS servers
- Heimdall timezone / uid / gid if needed
- WG-Easy hostname/domain
- WG-Easy admin credentials
- WireGuard DNS setting
- WireGuard subnet
- watchtower schedule settings if configurable
- SSH port if non-default
- service ports if user wants custom ones

Overall objective:
Create a ready-to-run deployment system under:
~/opt/deployment/

Required directory structure:
~/opt/deployment/
├── compose/
├── env/
├── data/
├── scripts/
└── logs/

For each app:
- compose/<app-name>/<app-name>.yaml
- env/<app-name>/<app-name>.env
- data/<app-name>/

Apps to prepare and deploy in this order:
1. DuckDNS
2. Heimdall
3. Pi-hole
4. WG-Easy

After that:
6. create maintenance scripts

8. configure weekly container updates

Critical constraints:
- Use Docker Compose
- Use one compose YAML per app
- Use one env file per app
- Use placeholders only through env files
- The YAML must reference env values and must not hardcode changeable values
- Persistent data must go under ~/opt/deployment/data/<app-name>/
- Do not start containers until the preparation phase is complete
- After preparation is complete, deploy one container at a time
- After each deployment, verify it is running
- If verification fails, stop and report the problem clearly

Version rule:
- For each app, determine the latest stable image tag at implementation time
- Verify the image tag from the official image source or official documentation
- Do not guess image tags
- Avoid using "latest" unless the official project explicitly recommends it
- The chosen tag must be written into the generated YAML so the deployment is reproducible

The installer must be interactive.

Implementation phases:

PHASE 1 - host preparation
The installer must:
1. update the Pi OS
2. install Docker Engine
3. install Docker Compose plugin
4. configure Docker to start on boot
5. configure Docker log rotation globally to prevent logs from filling the SD card
6. create the deployment directory structure
7. create a logs directory for installer and maintenance logs

PHASE 2 - file generation
For each app, in the required order, the installer must:
1. create the compose/<app-name>/ directory
2. create the env/<app-name>/ directory
3. create the data/<app-name>/ directory
4. ask the user for the values needed for that app
5. write the <app-name>.env file
6. write the <app-name>.yaml file
7. do not deploy yet
8. validate that required values were provided before moving on

PHASE 3 - maintenance scripts
The installer must create scripts under:
~/opt/deployment/scripts/

Required scripts:
1. weekly OS update script
2. Docker cleanup script
3. optional deployment verification helper if useful
4. firewall hardening script

The weekly OS update script must:
- run apt update
- run apt full-upgrade or equivalent safe OS update command
- remove obsolete packages
- log output to ~/opt/deployment/logs/
- be designed to run weekly at 4:00 AM

The Docker cleanup script must:
- remove unused images/containers/networks/build cache
- not remove Docker volumes automatically
- log output to ~/opt/deployment/logs/

PHASE 4 - scheduling
The installer must set up scheduled tasks so that:
1. weekly OS update runs first at 4:00 AM
2. Watchtower runs after the OS update
3. Docker cleanup runs after updates
4. all scheduled tasks log output

Use a reliable scheduler available on Raspberry Pi OS.
Cron is acceptable.

PHASE 5 - deployment
Only after all files and scripts are prepared:
1. deploy DuckDNS
2. verify DuckDNS is running
3. deploy Heimdall
4. verify Heimdall is running
5. deploy Pi-hole
6. verify Pi-hole is running
7. deploy WG-Easy
8. verify WG-Easy is running
9. deploy Watchtower
10. verify Watchtower is running

Deployment rule:
Each app must be deployed individually using its own compose file and env file.

Verification rule:
For each app, verify by checking at minimum:
- Docker container is running
- Docker compose reports healthy/running state if available
- published port is listening where applicable

If a service fails:

- print the failure clearly
